## 问题

编写和维护 CRUD 后端代码是重复的样板工作。当 LLM 每次都从头生成时，输出会漂移——不同的路由模式、不一致的请求体结构、特制的 SQLite 连接、遗漏的迁移。后端与前端的合约产生分歧，每一次重新生成都可能引入微妙的破坏。

Axle 以描述优先的方式解决这个问题：用 JSON 定义资源，生成确定性的 Go 代码，挂载为 HTTP 处理器，让 `axle check` 在生产前捕获漂移。

## 特性

- **描述优先的代码生成** — 在 JSON 中定义字段、CRUD 操作和自定义动作。Axle 从单个描述文件生成 Go 类型、路由元数据、注册表、OpenAPI 规范和 SQL 迁移。
- **多资源目录** — 通过简单的 JSON 清单将任意数量的资源描述组合成一个统一目录。该目录将所有资源挂载在单个 HTTP 处理器后面。
- **具体的 SQLite CRUD** — 没有 ORM，没有查询构建器，没有数据库抽象。一个纯 Go SQLite 门面为每个生成的资源处理列表、获取、创建、更新和删除操作。
- **确定性、不可触碰的输出** — 生成的文件带有 `DO NOT EDIT` 头部，相同描述下字节一致。没有反射、没有运行时目录扫描、没有注册魔法。
- **架构防膨胀检查** — `axle check` 拒绝手写的 CRUD 路由、通用数据库包装器、类型化仓库、反射发现和公共 API 膨胀——这些正是 LLM 默认倾向的模式。
- **后端脚手架** — `axle app init` 生成一个完整、可验证的 Go 后端骨架，包含生成的 CRUD、迁移、验证脚本，以及 Axle 拥有的生成代码与应用拥有的处理器之间的清晰所有权分割。
- **边缘运行时便利** — 可选的 `NewEdge` 包装器添加 `/healthz`、`/routes`、CORS 和 API 前缀规范化，无需手写路由开关。

## 何时使用

当你需要一个通过 AI 辅助重新生成来维护的 Go/SQLite CRUD 后端时，请使用 Axle。在定义好资源模型后使用，而非在初始原型阶段。

**不适用于：** 多数据库支持、管理 UI 生成、类型化 ORM 层、基于反射的注册或运行时目录扫描。Axle V1 有意限定在单进程 SQLite 后端加上描述驱动的代码生成。

## 快速开始

```bash
# 创建一个新的后端骨架
go run github.com/cosmo-wise/axle/cmd/axle app init \
  --out /tmp/axle-backend \
  --module example.com/axle-backend \
  --axle-replace /path/to/axle

# 验证它是否工作
cd /tmp/axle-backend && ./scripts/verify.sh
```

## 安装

```bash
go install github.com/cosmo-wise/axle/cmd/axle@latest
```

需要 Go 1.24+。除了纯 Go SQLite 构建外，没有运行时依赖。

## 使用

### 核心循环

在 `descriptors/tasks/descriptor.axle.json` 中定义资源，然后生成：

```bash
axle gen --descriptor descriptors/tasks/descriptor.axle.json --out descriptors/tasks/generated
```

输出包括 `types.gen.go`、`routes.gen.go`、`registry.gen.go`、`openapi.gen.json` 和迁移 SQL。

检查生成的输出而不实际写入：

```bash
axle gen --descriptor descriptors/tasks/descriptor.axle.json --out descriptors/tasks/generated --check --json
```

### 多资源目录

对于有两个或更多资源的应用，创建 `catalog/axle.catalog.json`：

```json
{
  "package": "catalog",
  "resources": [
    {"alias": "plans", "import": "example.com/app/descriptors/plans/generated"},
    {"alias": "tasks", "import": "example.com/app/descriptors/tasks/generated"}
  ]
}
```

```bash
axle catalog gen --manifest catalog/axle.catalog.json --out catalog
```

### 挂载为 HTTP 服务器

```go
db, _ := axlesqlite.Open(ctx, ":memory:", appcatalog.Catalog)
db.Migrate(ctx)
handler := axleruntime.New(appcatalog.Catalog, db, axleruntime.ActionHandlers{
    resources.HandlerRenameResource: resources.BindRenameResource(renameResource),
})
```

或者使用边缘运行时便利功能：

```go
handler := axleruntime.NewEdge(appcatalog.Catalog, db, handlers, axleruntime.EdgeOptions{
    Name:      "Task backend",
    APIPrefix: "/api/v1",
    CORS:      true,
})
```

### 运行架构检查

```bash
axle check --root . --json
```

这会拒绝手写 CRUD 路由、数据库抽象、反射、过期的生成文件和其他常见的 LLM 漂移模式。

### 验证就绪状态

```bash
axle doctor --json
```

报告 Go 二进制可用性和 SQLite 运行时健康状态。

## 项目边界

- `cmd/axle` — 薄 CLI 边缘
- `internal/*` — 描述解析、代码生成、OpenAPI、检查、SQLite 内部
- `pkg/axle` — 仅元数据的公共合约（Descriptor、Catalog、ResourceRegistry）
- `pkg/axle/runtime` — 生成目录的 HTTP 挂载
- `pkg/axle/sqlite` — 具体的 SQLite CRUD 门面

生成的文件带有 `Code generated by axle gen; DO NOT EDIT.` 标记。修改描述文件或目录清单，然后重新生成。

## 工作原理

Axle 读取 JSON 资源描述，根据版本化模式进行解析，然后确定性生成 Go 源文件、OpenAPI 规范和 SQL 迁移。生成的类型包括资源 DTO、CRUD DTO、类型化动作绑定器和路由元数据。目录清单将多个生成的包组合成一个注册表。运行时将传入的 HTTP 请求与目录路由元数据匹配，并分发到 SQLite 门面或已注册的自定义动作处理器。

## 部署

Axle 后端是标准的 Go 二进制文件。使用 `go build` 构建并部署为单个二进制文件。SQLite 数据库文件路径在运行时指定——没有嵌入式服务器或外部数据库进程。
